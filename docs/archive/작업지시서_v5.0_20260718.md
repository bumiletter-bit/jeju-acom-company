### 0-1. 버전 체계 확립
- 현재 배포 상태를 **v4.0**으로 공식 명명하고 시작점으로 태깅한다 (이전 15회 배포는 v1~v3.x로 소급 기록만).
- 이 지시서의 모든 작업이 완료된 상태 = **v5.0**.
- 규칙: 기능 추가 = 0.1 단위 증가(v5.1, v5.2...), 대형 개편 = 정수 증가(v6.0), 핫픽스 = v5.0.1 형태.

### 0-2. 구현 항목
1. **VERSION 상수**: server.js 또는 별도 version.js에 현재 버전 문자열 관리. API `/api/version`으로 노출.
2. **화면 표시**: AGENT OFFICE 화면 하단(또는 사이드바)에 `v5.0` 항상 표시. 대표가 지금 어떤 버전을 쓰는지 즉시 확인 가능하게.
3. **CHANGELOG.md 생성**: 버전별로 다음 형식으로 기록. 이후 모든 배포 시 갱신 필수.
   ```
   ## v5.0 (2026-07-17)
   - [수정] 월 단위 날짜 정규식 확장 (2025-04 오해석 사고 재발 방지)
   - [추가] 조회 조건부 복창 확인
   - [백업] backup_v4.0_20260717.sql
   ```
4. **배포 전 백업 자동화**: 배포 스크립트에 ①git tag(v버전) ②DB 덤프(pg_dump → `backup_v{버전}_{날짜}.sql`, 외부 저장 위치에 보관) 를 포함. 백업 없이는 배포 불가하도록 스크립트에서 강제.
5. **롤백 절차 문서화 + 간편화**: 오류 발생 시 "직전 안정 버전으로 복귀"를 명령 1~2개로 실행 가능하게.
   - 코드: `git checkout v{이전버전}` → 재배포
   - DB: 해당 버전 백업 파일로 복원 (단, 롤백 시 DB 복원은 데이터 유실 가능성이 있으므로 "코드만 롤백"과 "코드+DB 롤백"을 구분해 문서화하고, 기본은 코드만 롤백)
   - CHANGELOG에 롤백 이력도 기록 ("v5.1 → v5.0 롤백, 사유: ...")
6. **버전-백업 매칭표**: CHANGELOG 하단에 버전별 백업 파일명 목록 유지. 대표가 "v4.0으로 돌려줘"라고 하면 헷갈림 없이 해당 백업을 특정할 수 있어야 한다.

## 1단계. 사고 재발 방지 패치 (2026-07-17 스크린샷 사고 5건 대응)

### 1-1. 월 단위 날짜 정규식 확장 🔴 최우선
- 현재 버그3 수정은 "N월 N일" 패턴만 커버 → "4월달 매출현황" 같은 **월 단위 지시**가 사각지대 (실제 2025-04 오해석 사고 발생).
- 서버 정규식 덮어쓰기를 다음 패턴까지 확장: `N월`, `N월달`, `지난달`, `이번달`, `저번달`, `YYYY년 N월`.
- 연도 생략 시 규칙: **가장 가까운 과거의 해당 월** (7월에 "4월" = 2026-04, 7월에 "9월" = 2025-09). 기존 "올해/미래면 작년" 로직이 월 단위에 미적용 또는 오작동했는지 원인 확인 후 수정.
- 검증: 사고 재현 케이스 포함 테스트 — "4월달 매출현황"(→2026-04), "지난달 정산"(→2026-06), "9월 매출"(→2025-09), "2025년 4월"(→2025-04, 명시 연도 존중) 등 최소 8케이스 ALL PASS.

### 1-2. 조회 조건부 복창 확인
- 등록·저장에만 있는 "확인 1회"를 조회에도 조건부로 적용.
- 발동 조건: 해석된 조회 기간이 **오늘 기준 3개월 이상 과거**이거나, **연도가 작년 이전**일 때.
- 동작: 실행 전 "『2025년 4월(04-01~04-30)』 조회로 진행할까요?" 복창 → 대표 승인(기존 정규식 판별 재사용) 후 실행.
- 최근 기간(이번주/이번달 등) 조회는 기존대로 즉답 (속도 유지).

### 1-3. 0원·0건 결과 가드
- 조회 결과가 0건·0원이면 그대로 보고하지 않고: "『2025년 4월』 조회 결과 0건입니다 — 조회 기간이 맞나요? (데이터가 있는 가장 가까운 달: 2026-04)" 형태로 조건 복창 + 인접 데이터 힌트 제공.
- 순수 코드(세미)에 구현 (0원). 정직 원칙 유지 — 다른 기간 데이터로 대체 금지는 그대로.

### 1-4. is_test 플래그 (테스트 격리)
- 역량 테스트 실행분에 `is_test=true` 부여.
- 보고서함·피드백·교훈 추출·오배정 통계에서 전부 제외 (자기오염 루프 차단).
- 테스트 결과는 별도 성적표 화면에서만 조회.

### 1-5. 마루 오배정 카운트 위젯

- 향후 모델 판단 근거 데이터로 활용 (감이 아닌 숫자로).

## 3단계. 일정 기능 확장 — 마케팅·운영 일정 통합

### 3-1. 일정 카테고리 도입
- 기존 마루 일정 처리(조회 즉답/등록 확인 1회)에 **카테고리** 추가:
  - `휴가` (담당자 필수 — 실제 직원 명단 매칭 기존 로직 재사용)
  - `톡톡발송` / `문자발송` (발송 예정 마케팅 일정)
  - `할인·이벤트` (할인 기간, 프로모션)
  - `일반` (기존 일정)
- 마루가 지시 문장에서 카테고리 자동 판별: "다음주 화요일 채연이 휴가" → 휴가, "토요일에 카라향 할인 문자 발송" → 문자발송, "25~27일 하우스감귤 오픈 할인" → 할인·이벤트(기간형).

### 3-2. 동작 규칙
- 등록 확인 표에 카테고리 표시: `날짜(요일) 시간 — [문자발송] 카라향 할인 안내 (담당: 대표)`
- 캘린더/일정 조회 시 카테고리별 아이콘·색상 구분 표시.
- **발송 일정 등록 ≠ 자동 발송**: 톡톡·문자 일정은 "예정 알림"일 뿐, 실제 발송은 기존 원칙(외부 행동 승인제) 그대로 대표가 수동 실행. 자동 발송 경로는 만들지 않는다.
- 발송·할인 일정은 당일(또는 전날) LIVE 로그/위젯에 "오늘 예정: [문자발송] 카라향 할인" 리마인드 표시.
- 할인 일정은 글샘 호출 시 컨텍스트로 주입 (예: 활성 할인 기간 중이면 카피에 해당 명분 활용 가능) — 단, 지식 문서의 "명분 없는 할인 표현 금지" 규칙과 충돌하지 않게 활성 할인만 전달.

## 4단계. 파일 전달 기능 — "이번주 결제금액 보내줘"

### 4-1. 보고서 파일 생성 (세미 확장, 순수 코드)
- 정산·매출 조회 결과를 **엑셀(xlsx) 파일**로 생성하는 기능 추가 (exceljs 등 서버 라이브러리 사용).
- 파일 구성: 시트1=요약(기간, 총액, 거래처별 합계, 택배), 시트2=상세(일자별·품목별 내역). 정산관리 화면 숫자와 1원 오차 없어야 함 (기존 검증 기준 유지).
- 파일명 규칙: `정산보고_{기간}_{생성일}.xlsx` (예: `정산보고_2026-07-13~07-17_20260717.xlsx`).

### 4-2. 지시 처리 흐름
- 마루가 "보내줘/파일로/엑셀로/다운로드" 류 키워드 감지 시 파일 생성 모드로 배정.
- "이번주 결제금액 보내줘" → 세미가 이번주 정산 집계 → xlsx 생성 → 보고서함에 **[📎 다운로드]** 버튼 첨부 + LIVE 로그에 완료 표시.
- 파일은 서버 저장 + adminOnly 다운로드 API로만 접근 (URL 직접 접근 시 401/403). 생성 이력 audit_log 기록.
- 보관 정책: 파일도 soft-delete 원칙 — 보고서함 [✔확인] 시 숨김만, 삭제 없음. 저장 용량 관리를 위해 90일 경과 파일은 재생성 가능하므로 물리 정리 허용(정리 이력 기록).

### 4.5-1. ⑤기간 비교 보고서
- 지시 예: "4월 5월 매출 비교해줘", "이번달 지난달 비교", "4월이랑 작년 4월 비교"
- 마루 라우팅에 **비교 의도 인식** 추가: "비교/대비/vs/차이" 키워드 + 기간 2개 추출. 기간 추출은 1단계 월 단위 정규식을 재사용(두 기간 모두 서버 정규식으로 확정).
- 출력 구성 (모두 코드 계산):
  - 총액 비교: 기간A vs 기간B 총액·증감액·증감률
  - 거래처별 비교: 거래처별 금액과 증감률
  - 품목별 비교: 두 기간 품목 TOP 10 나란히 + 증감 표시, 한쪽에만 있는 품목은 [신규]/[종료] 태그
  - 건수·택배 박스 비교
- 정직 원칙 유지: 한쪽 기간 데이터 없으면 억지 % 없이 "데이터 없음" 표시 (기존 전년 동기 규칙과 동일).

### 4.5-2. ⑥품목 매출 기여 순위
- 지시 예: "매출기여 가장 큰 품목 뭐야", "4월 품목 순위", "잘 팔린 거 TOP 5"
- 출력: 지정 기간(미지정 시 이번달)의 품목별 **판매수량·매출액·전체 대비 비중(%)** 순위표. 기본 TOP 10, "전부/전체" 요청 시 전 품목.
- 동일 품목의 규격 다른 상품(예: 카라향 3kg/5kg/9kg)은 개별 순위 + **계열 합계**(카라향 전체) 별도 줄 제공 — 품목 계열 매칭은 기존 토큰 분해 매칭 로직 재사용("하귤" 제외 규칙 포함).
- 단가·금액은 기존 3단계 매칭 그대로, 정산관리 화면과 1원 오차 없어야 함.

## 5단계. 보안·안전성 종합 점검 (v5.0 배포 전 필수)

1. **권한 재검증**: 신규 API 전부(버전, 파일 다운로드, 일정 카테고리) adminOnly 적용 + 무인증/위조 토큰 401·403 테스트.
2. **DB 백업 체계 확인**: Render DB 자동 백업 정책 확인 + 0단계 버전별 백업과 별개로 **주기 백업(최소 주 1회) 외부 보관** 설정. soft-delete는 백업이 아님을 전제.
3. **MCP 시크릿 점검**: URL 시크릿 유출 대비 로테이션(교체) 절차를 문서로 남김. 코드·로그·화면 어디에도 시크릿 전체 문자열이 출력되지 않는지 스캔.
4. **학습 루프 안전장치**: 요원당 활성 노트 상한(10개) + 교훈별 [롤백] 버튼 + 롤백 이력 기록. 교훈 승인 권한은 대표 전용으로 코드 레벨 고정 (추후 팀 공유 시에도 불변).
5. **기존 원칙 회귀 확인**: 허위보고 방지(로그는 코드 기록), 정직 원칙(폴백 금지), 외부 행동 승인제, 삭제 없음 — 이번 작업으로 훼손된 곳 없는지 전수 확인.
6. **역량 테스트 전 문항 통과** 후에만 v5.0 태그 + 배포.
