# 지시함 운영 규칙 (지시 #25 — 2026-07-19 명문화)

> 지시함(cc_instructions) 파이프라인의 상태 전환·경과 보고 규칙.
> 텔레그램 알림: 상태 변화 = "보고 도착" / 상태 동일 + 응답 갱신 = "경과 갱신" (b안, 대표 결정).

## 상태 전환 규칙

1. **완료 전환을 외부 대기 뒤에 묶지 말 것** — 빌드·부팅·대표 클릭 등 외부 대기가 남았어도, 지시의 본 작업이 끝났으면 그 사실을 먼저 기록한다. 외부 대기 결과 확인이 지시의 요건이면 "진행 + 대기 중 경과"로 명시.
2. **긴 대기 발생 시 "대기 중" 경과 갱신을 먼저 남길 것** — 응답 갱신이 텔레그램 "경과 갱신" 알림을 발생시키므로, 대표가 폰에서 대기 상황을 인지할 수 있다.
3. **대기와 무관한 지시(통보 등)는 즉시 완료 전환.**
4. **30분 이상 무갱신 금지** — 장기 작업은 최소 30분 간격으로 경과를 기록한다.

## 똑똑이 실행 도구 (v5.1.1 지시 #28-2 → 지시 #51로 2종 확대)

- MCP 실행 도구는 **2종만** 허용: `run_capability_test`(역량 점검) · `run_smoke_test`(실전 스모크 — 지시 #51). 그 외 실행·쓰기 불가, 읽기 전용 관측 3종과 분리.
- `run_smoke_test`는 [#51-실행] 지시 등록 + 대표 'ㄱ' 승인 후에만 호출 (AI 약 9회 비용). 전 문항 is_test 격리 — 실제 등록·발송·생성 없음. audit `mcp_run_smoke` 전수 기록.
- **승인 게이트 (대표 확정)**: 똑똑이는 대화에서 대표 승인("ㄱ")을 받은 후에만 호출한다. 기계적 상한은 없음.
- 서버는 호출 전수를 audit(`mcp_run_test`)로 기록 — 시작 실패(중복 실행 등)여도 호출 시도 자체가 남아 사후 감사 가능.
- 결과: 완료 시 텔레그램 발송(기존 유지), 문항 상세는 `get_test_results`로 조회.

## 역량 점검 실행 정책 — 묶음 게이트 (지시 #40)

- 점검 실행은 **버전 태그 직전 1회** (지시 1건마다 실행 금지 — 회귀 합산 표준). 예외 = 대표 명시 요청(사고 수정 검증 등)뿐.
- 개별 지시 완료 판정은 코드·로컬 검증으로, 보고에 "역량 회귀는 다음 태그 게이트에 합산" 기본 문구 사용.
- 게이트 실행은 똑똑이 run_capability_test(대표 'ㄱ' 승인)로 — 대표는 승인만.

## 기존 원칙 (재확인)

- 지시함은 전달 통로일 뿐 권한 확대 아님 — 외부 행동 승인제·자동수정 금지·보고 우선 원칙 동일 적용.
- 완료 전환은 응답(작업 결과) 기록과 함께만 가능 (`respond-instruction.js <id> 완료 <응답파일>`).
- 검수 프로토콜 v2: 기술 검증은 클로드 코드 자체 전담, 완료 시 대표 스모크 1건 지정. 판독은 똑똑이 관측 도구(get_live_log/get_test_results/get_reports) 활용.
- 배포 후 반드시 부팅 selfcheck audit 신규 행 확인 — Render push 웹훅 간헐 유실(7/19 2회) 대응, 미부팅 시 빈 커밋 재push.
